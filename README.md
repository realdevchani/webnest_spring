# 프로젝트 개요

본 프로젝트는 코리아IT아카데미 강남점 풀스택 개발자 양성과정에서 진행한 5인 팀 프로젝트입니다.  
실제 협업 및 개발은 팀 원본 저장소에서 진행되었으며, 본 저장소는 포트폴리오 제출을 위해 제가 담당한 내용을 중심으로 정리한 레포지토리입니다.

- 팀 원본 저장소: https://github.com/NullPoint-team/webnest_spring.git

# 프로젝트 소개

- **프로젝트명**: WebNest
- **제작 기간**: 2025.07.29 ~ 2025.11.29
- **참여 인원**: 5명
- **주제**: 10대를 위한 웹 프로그래밍 커뮤니티 서비스

## 제작 배경
코딩 및 프로그래밍 교육의 중요성이 점차 커지는 흐름 속에서,  
프로그래밍을 처음 접하는 10대 학생들이 또래의 언어로 질문하고 답변을 주고받으며 자연스럽게 학습할 수 있는 서비스가 필요하다고 생각했습니다.  
또한 미니 게임 요소를 함께 제공하여, 프로그래밍에 대한 흥미를 보다 쉽게 느낄 수 있는 웹 서비스를 기획하게 되었습니다.

# 주요 기능

## 1. 검색
- 게시글 제목과 내용을 대상으로 검색 기능 구현
- DB에서는 검색 후보군을 조회하고, 서비스 계층에서 점수 계산 후 정렬
- 제목/내용 가중치를 다르게 적용하여 검색 품질 개선

## 2. 알림
- 내 게시글, 댓글에 새로운 상태 변경(좋아요, 새 댓글, 답글) 시 페이지 HEADER 에 알림상태 반영
- 알림 목록 조회 및 상태 반영 처리

## 3. 게임방리스트
- 게임방 생성, 게임방 상태 변경처리
- 게임방 입장 시 비밀방 비밀번호 인증처리

## 4. 멀티 게임(주사위 놀이)
- 주사위 굴린 후 event 발생 시 해당 위치로 이동 처리
- 주사위 굴린 후 차례 정보 넘기기
- 주사위 2개 일치 시 차례 정보 넘기지 않고, 주사위 다시 굴리기 처리

## 5. 채팅
- 게임방 내부 사용자 간 실시간 채팅 기능 구현
- 채팅 메시지 송수신 및 화면 반영 처리

# 트러블슈팅

### 1. 게임방 퇴장 및 연결 종료 상황에서 방 생명주기 관리 개선

- **기존 방식**
  - 사용자가 게임방을 나가더라도 방 정보가 즉시 정리되지 않아, 빈 게임방이 목록에 남는 문제가 있었습니다.
  - 또한 새로고침이나 비정상 종료가 발생했을 때도 서버에서 참여 상태가 바로 정리되지 않아, 실제로는 이탈한 사용자가 방에 남아 있는 것처럼 보이는 상태가 발생했습니다.

- **문제점**
  - 실제 사용자 상태와 서버에 저장된 게임방 정보가 일치하지 않아, 게임 진행 중 상태 꼬임이 발생할 수 있었습니다.
  - 특히 새로고침은 완전한 퇴장이라기보다 일시적인 재접속 상황일 수 있어, 무조건 즉시 삭제하는 방식은 사용자 경험 측면에서도 한계가 있었습니다.

- **개선 방식**
  - 사용자가 명시적으로 나가기 버튼을 눌러 퇴장한 경우에는 즉시 방 참여 상태를 정리하도록 처리했습니다.
  - 반면 새로고침이나 비정상 종료처럼 일시적 연결 끊김 가능성이 있는 경우에는 TTL 기반 유예 시간을 두고, 일정 시간 내 재접속이 없을 때 최종 퇴장 및 방 삭제가 이루어지도록 방 생명주기 관리 방식을 보완했습니다.
  - 마지막 사용자가 최종 이탈한 경우에는 게임방 자체를 삭제하도록 하여, 빈 방이 지속적으로 남지 않도록 개선했습니다.

- **결과**
  - 빈 게임방이나 유령 사용자 상태가 남는 문제를 완화할 수 있었습니다.
  - 게임방 생성 → 입장 → 일시적 연결 종료 → 재접속 또는 최종 이탈 → 방 삭제까지의 흐름을 더 안정적으로 관리할 수 있게 되었습니다.

### 2. 주사위 게임 턴 로직 개선

- **기존 방식**
  - 초기에는 개인전 기준으로, 게임방에 입장한 사용자 순서대로 턴이 순차적으로 돌아가도록 구현했습니다.

- **문제점**
  - 이후 게임 규칙이 개인전에서 팀전으로 변경되면서, 단순 입장 순서 기반 로직으로는 팀전 진행 방식을 반영하기 어려웠습니다.
  - 같은 팀 사용자가 연속으로 턴을 가져가거나, 팀 간 번갈아 진행되어야 하는 흐름이 제대로 반영되지 않는 문제가 있었습니다.

- **개선 방식**
  - 사용자 정보를 팀 색상 기준으로 분류한 뒤, 각 팀이 번갈아 턴을 가져갈 수 있도록 순서를 재구성했습니다.
  - 기존의 단순 순차 처리 방식에서 벗어나, 팀전 규칙에 맞게 색상 기준으로 퐁당퐁당 턴이 진행되도록 로직을 수정했습니다.

- **결과**
  - 팀전 규칙에 맞는 턴 흐름을 반영할 수 있게 되었고, 게임 진행 방식이 기획 의도와 일치하도록 개선했습니다.
  - 게임 규칙 변경에 맞춰 기존 로직을 재설계하고 확장하는 경험을 얻을 수 있었습니다.

## 3. 검색 기능 개선

- **기존 방식**
  - 초기에는 SQL `LIKE '%검색어%'` 기반의 단순 포함 검색을 사용했습니다.
  - 이 방식은 검색어가 제목이나 내용에 포함되기만 하면 동일한 우선순위로 조회되어,
    사용자가 원하는 게시글이 상단에 노출되지 않는 문제가 있었습니다.

- **문제점**
  - 제목에 검색어가 직접 포함된 게시글과, 본문에 우연히 포함된 게시글이 동일한 우선순위로 조회되었습니다.
  - 단순 포함 여부만으로는 사용자의 검색 의도에 더 가까운 게시글을 먼저 보여주기 어려웠습니다.
  - 점수 기준이 복잡해질수록 SQL만으로 검색 우선순위를 관리하는 방식은 유지보수성이 떨어질 수 있다고 판단했습니다.

- **개선 방식**
  - 데이터베이스에서는 제목 또는 내용에 검색어가 포함된 게시글을 **후보군으로 먼저 조회**하도록 구성했습니다.
  - 이후 서비스 계층(Java)에서 검색 결과에 점수를 부여하는 방식으로 검색 로직을 개선했습니다.
  - 점수 기준은 다음과 같이 설정했습니다.
    - 제목에 검색어 전체 포함: 10점
    - 내용에 검색어 전체 포함: 5점
    - 제목에 검색어 키워드 일부 포함: 2점
    - 내용에 검색어 키워드 일부 포함: 1점
  - 최종 점수를 기준으로 내림차순 정렬하고, 점수가 같은 경우 최신 게시글이 먼저 노출되도록 처리했습니다.

- **선택 이유**
  - 검색 후보군 조회는 DB가 담당하고, 최종 랭킹 계산은 Java 서비스 계층이 담당하도록 역할을 분리했습니다.
  - 이를 통해 SQL을 과도하게 복잡하게 만들지 않으면서도, 검색 기준 변경에 유연하게 대응할 수 있도록 설계했습니다.

- **결과**
  - 단순 문자열 포함 검색에서 벗어나, 사용자의 검색 의도에 더 가까운 게시글이 상위에 노출되도록 개선했습니다.
  - 검색 점수 기준을 코드에서 명확하게 관리할 수 있어 유지보수성과 확장성을 높일 수 있었습니다.

# 배운 점

## 1. **협업에서는 일정 관리가 개발 생산성에 직접적인 영향을 준다는 점을 배웠습니다.**  
   검색 기능 구현 과정에서 선행되어야 할 화면 작업이 지연되며 개발 흐름에 영향을 받은 경험이 있었습니다.  
   이를 통해 협업에서는 맡은 기능을 구현하는 능력뿐 아니라, 기한 내에 결과물을 공유하는 책임감이 중요하다는 점을 배웠습니다.

## 2. **기능 명세와 실제 구현 내용을 연결하는 문서화의 중요성을 배웠습니다.**  
   구현한 기능이 명세서대로 동작하는지 확인하고, 이를 팀원들과 공유하기 위해서는 관련 내용을 정리하는 과정이 필요했습니다.  
   이를 통해 문서는 단순한 기록이 아니라, 개발 방향과 결과를 맞추는 기준이라는 점을 배웠습니다.

## 3. **협업 코드에서는 개인 스타일보다 일관된 규칙이 더 중요하다는 점을 배웠습니다.**  
   코드 재사용과 유지보수를 고려했을 때, 각자 다른 방식으로 작성된 코드보다 팀 내에서 합의된 네이밍, 구조, 작성 규칙이 있는 코드가 훨씬 관리하기 쉽다는 점을 느꼈습니다.  
   이를 통해 협업 환경에서는 통일된 코드 컨벤션이 생산성과 유지보수성에 큰 영향을 준다는 점을 배웠습니다.
